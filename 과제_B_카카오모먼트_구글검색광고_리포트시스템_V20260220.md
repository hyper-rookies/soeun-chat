# 과제 B: 카카오 모먼트 + 구글 검색광고 AI 리포트 시스템

> 생성일시: 2026-02-20 KST  
> 과제 유형: 개인 과제 (1차 - 리포트 기능)  
> 수행 기간: 2주

---

## 1. 과제 개요

광고주의 **카카오 모먼트** 디스플레이 광고와 **구글 검색광고(Google Ads)** 데이터를 수집하여, 자연어 대화 인터페이스로 성과를 조회하고 분석할 수 있는 AI 리포트 시스템을 구축한다.

사용자는 복잡한 대시보드를 직접 조작하지 않고, 채팅창에 질문을 입력하면 AI가 데이터를 조회하고 차트와 표로 답변한다.

**핵심 가치:** "대시보드를 보지 말고, 대화하세요."

## 2. 광고 개념 학습 목표

과제를 수행하기 전에 아래 개념을 반드시 이해하고 시작할 것.

### 2.1 구글 검색광고(Google Ads) 기본 개념

| 용어 | 설명 |
|------|------|
| 캠페인 (Campaign) | 광고의 최상위 단위. 예산, 타겟 지역, 일정 설정 |
| 광고그룹 (Ad Group) | 캠페인 하위 단위. 키워드와 광고 소재 묶음 |
| 반응형 검색광고 (RSA) | 제목/설명 여러 개를 자동 조합하는 광고 형식 |
| 품질 점수 (Quality Score) | 키워드 관련성, 클릭률, 랜딩페이지 품질 종합 점수 (1~10) |
| 광고 순위 (Ad Rank) | 입찰가 × 품질점수로 결정되는 노출 위치 |
| 노출 (Impression) | 광고가 화면에 표시된 횟수 |
| 클릭 (Click) | 사용자가 광고를 클릭한 횟수 |
| CTR | Click-Through Rate = 클릭수 / 노출수 × 100 |
| CPC | Cost Per Click = 총 비용 / 클릭수 |
| 전환 (Conversion) | 광고 클릭 후 목표 행동 완료 (구매, 신청 등) |
| 전환율 (CVR) | 전환수 / 클릭수 × 100 |
| CPA | Cost Per Action = 총 비용 / 전환수 |
| ROAS | Return On Ad Spend = 전환매출 / 광고비 × 100 |
| 스마트 입찰 | Google AI가 전환 목표에 맞게 자동 입찰 조정 |

### 2.2 카카오 모먼트 기본 개념

| 용어 | 설명 |
|------|------|
| 광고 계정 | 카카오 비즈니스 광고 계정 단위 |
| 캠페인 | 광고 목적 단위 (전환, 방문, 도달 등) |
| 광고그룹 | 타겟, 일정, 예산 설정 단위 |
| 광고 소재 | 실제 노출되는 이미지/문구 |
| 카카오 픽셀 & SDK | 전환 추적을 위한 웹/앱 추적 코드 |
| 도달 (Reach) | 광고를 본 순 사용자 수 |
| 노출 (Impression) | 광고 노출 총 횟수 |
| CPM | Cost Per Mille = 1,000회 노출당 비용 |
| 클릭 (Click) | 광고 클릭 횟수 |
| CPC | Cost Per Click = 총 비용 / 클릭수 |
| 전환 (Conversion) | 픽셀/SDK로 추적된 목표 행동 |

### 2.3 공통 리포트 지표 계산

```
CTR(%) = 클릭수 / 노출수 × 100
CPC(원) = 총비용 / 클릭수
CVR(%) = 전환수 / 클릭수 × 100
CPA(원) = 총비용 / 전환수
ROAS(%) = 전환매출 / 총비용 × 100
CPM(원) = 총비용 / 노출수 × 1000
```

### 2.4 검색광고 vs 디스플레이 광고 차이

| 구분 | 검색광고 (구글SA) | 디스플레이 광고 (카카오 모먼트) |
|------|-----------|-----------| 
| 노출 방식 | 사용자가 검색할 때 | 카카오 앱/서비스 내 배너 |
| 타겟팅 | 키워드 기반 | 관심사/행동/인구통계 기반 |
| 주요 지표 | CPC, CTR, 전환율 | CPM, 도달, 빈도 |
| 특성 | 구매 의도 높음 | 브랜드 인지도/재타겟팅 |

## 3. LLM 기반 리포트란 무엇인가

### 3.1 기존 방식 vs LLM 방식

기존에 리포트 시스템을 개발할 때는 아래처럼 만들었다.

```
[기존 룰베이스 방식]
개발자가 직접 SQL 작성
  → 정해진 템플릿에 데이터 바인딩
  → HTML/PDF 렌더링
  → 사용자는 정해진 리포트만 볼 수 있음

문제점:
- "이번 달 ROAS 낮은 캠페인 TOP3만 뽑아줘" 같은
  요청이 오면 개발자가 새 기능을 만들어야 함
- 모든 경우의 수를 미리 개발해둬야 함
```

이 과제에서 만드는 방식은 다르다.

```
[LLM 기반 방식]
사용자가 자연어로 질문
  → Claude가 질문 의도 파악
  → Claude가 필요한 데이터 쿼리를 직접 생성
  → 데이터 조회 후 분석 + 시각화 형식 결정
  → 사용자에게 자연어 + 차트/표로 답변

장점:
- 미리 개발 안 해도 다양한 질문 처리 가능
- 사용자가 원하는 방식으로 데이터를 유연하게 조회
- 분석 인사이트까지 자동 생성
```

### 3.2 핵심 개념: Tool Use (도구 호출)

Claude가 데이터를 조회하는 방법은 "Tool Use" 패턴이다. Claude에게 "이런 도구가 있어"라고 알려주면, Claude가 스스로 판단해서 도구를 호출한다.

```
[흐름 예시]

사용자: "이번 달 구글 캠페인별 전환수 보여줘"

1. Claude가 질문 분석
   → "구글 캠페인 데이터, 기간은 이번 달"

2. Claude가 query_aurora 도구 호출 결정
   → SQL 자동 생성:
     SELECT campaign_name, conversions, cost,
            ROUND(cost/conversions, 0) as cpa
     FROM daily_campaign_summary
     WHERE platform = 'google_search'
       AND report_date >= '2026-02-01'
     ORDER BY conversions DESC

3. Lambda가 Aurora 실행 → 결과 반환

4. Claude가 결과 분석 + 응답 생성
   → "브랜드 캠페인 전환수 142건으로 1위..."
   → 바 차트 스펙 생성

5. 프론트엔드가 차트 렌더링
```

### 3.3 시스템 프롬프트의 역할

Claude에게 전달하는 시스템 프롬프트가 서비스의 핵심이다. 이 안에 Claude의 역할 정의, 사용 가능한 도구 목록, 응답 형식 규칙, 금지 사항이 모두 담긴다. **프롬프트 품질이 곧 서비스 품질이다.**

### 3.4 왜 SSE 스트리밍이 필요한가

Claude가 응답을 생성하는 데 5~10초가 걸린다. 이 시간 동안 화면이 멈춰있으면 사용자는 오류라고 느낀다. SSE(Server-Sent Events)는 생성된 텍스트를 실시간으로 글자 단위로 흘려보내는 기술이다. ChatGPT에서 답변이 타이핑되듯 나오는 것과 동일한 방식이다. 기존 HTTP 요청-응답 방식으로는 구현할 수 없고, 연결을 유지한 채 서버에서 데이터를 밀어넣는 단방향 스트리밍 구조다.

## 4. 기능 요구사항

### 4.1 데이터 수집 (배치 파이프라인)

**구글 검색광고 수집 항목**
- 계정 정보 (customer_id, 계정명)
- 캠페인 목록 및 일별 성과 (노출, 클릭, 비용, 전환, 전환매출)
- 광고그룹별 일별 성과
- 키워드별 일별 성과 (품질점수 포함)

**카카오 모먼트 수집 항목**
- 광고 계정 정보
- 캠페인 목록 및 일별 성과 (노출, 클릭, 비용, 도달, 전환)
- 광고그룹별 일별 성과

**배치 스케줄**
- 매일 새벽 3시 EventBridge 트리거
- 전일 데이터 수집 (D-1 기준)
- 실패 시 SQS 재시도 (최대 3회)
- 최종 실패 시 로그 기록

### 4.2 채팅 인터페이스

**레이아웃**
- ChatGPT 스타일 사이드바 + 대화창
- 사이드바: 대화 목록, 새 대화 버튼
- 대화창: 메시지 스트리밍 (SSE), 차트/표 렌더링

**지원 질의 유형 (1차)**
```
"이번 달 구글 캠페인별 전환수 보여줘"
"지난주 카카오 광고 CPM 추이 알려줘"
"어제 ROAS가 가장 높은 캠페인은?"
"2월 전체 매체별 비용 비교해줘"
"품질 점수 5점 이하 키워드 목록 보여줘"
"카카오 캠페인 중 클릭률이 낮은 광고그룹은?"
```

**응답 형식**
- 텍스트 분석 + 차트 (Recharts)
- 텍스트 분석 + 데이터 표 (정렬/필터 가능)
- 텍스트만

### 4.3 인증 및 계정 관리

- Google OAuth 로그인 (Cognito 연동)
- 광고 계정 연동 설정 페이지
- 구글 Ads API 인증 설정 (OAuth2)
- 카카오 모먼트 API 토큰 등록

### 4.4 기본 대시보드

채팅 외에 간단한 요약 대시보드 1페이지:
- 오늘/어제/이번 주 총 광고비
- 매체별 비용 비중 (파이 차트)
- 최근 7일 일별 전환 추이 (라인 차트)

## 5. 개발 방법론 - 바이브 코딩 가이드

### 5.1 기본 원칙

이 과제는 **100% AI 보조 개발(Vibe Coding)** 방식으로 진행한다. 직접 코드를 처음부터 작성하지 않고, Claude와 대화하면서 코드를 생성하고 이해하며 개선한다.

### 5.2 Claude 활용 방법

**기획 단계에서의 대화 방식**
```
나쁜 예: "코드 짜줘"
좋은 예: "구글 Ads API에서 캠페인 일별 성과를 GAQL로 조회해서
         Lambda에서 처리하고 S3에 Parquet으로 저장하려고 해.
         cost_micros를 원화로 변환하는 것도 포함해서
         전체 Lambda 코드를 TypeScript로 작성해줘."
```

**막혔을 때 질문 방법**
```
"아래 에러가 발생했어. 어떤 원인인지 설명해주고 해결 방법 알려줘:
[에러 메시지 붙여넣기]

현재 내 코드는 이래:
[코드 붙여넣기]"
```

**이해 확인 방법**
```
"GAQL이 일반 SQL이랑 다른 점이 뭐야?
 JOIN은 왜 안 되는지 설명해줘."
```

### 5.4 [2주차 Day 12-14] 자동 리포트 + 링크 공유 기능

**목표:** 매주 월요일 오전 8시에 지난 주 성과 리포트를 자동 생성하고, 외부 공유 가능한 링크를 만든다.

**구현 범위**

```
[자동 리포트 생성]
EventBridge (매주 월요일 08:00)
  → Lambda 트리거
  → Claude에게 "지난주 전체 성과 요약해줘" 자동 요청
  → 응답 결과를 DB에 저장 (report_id 생성)

[링크 공유]
저장된 리포트에 JWT 토큰 기반 공유 링크 생성
  → 링크 예시: https://도메인/shared/eyJhbGc...
  → 로그인 없이 접근 가능 (읽기 전용)
  → 만료일 설정 (기본 30일)
```

**구현 포인트**
- EventBridge cron 표현식: `cron(0 23 ? * SUN *)` (UTC 기준 일요일 23시 = KST 월요일 08시)
- JWT 생성: `jsonwebtoken` 라이브러리 사용
- 공유 페이지: Next.js의 `/shared/[token]` 라우트, 로그인 미들웨어 제외 처리

### 5.5 코드 이해 의무

AI가 생성한 코드라도 반드시 이해하고 넘어갈 것. 이해 안 되는 코드는 Claude에게 설명 요청.
